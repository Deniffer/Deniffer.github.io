记录做lazy Allocation的一些内容，

所谓lazy allocation就是指当用户程序想要申请堆上的内存的时候(malloc)，先不真正分配内存，而是假装分配了，其实并没有分配真正的物理页或者是虚存，等到page fault的时候再分配物理内存。 malloc底层调用的是system call sbrk()

实现过程

  • 其实实现过程也是很简单的，主要利用到了页表的机制（pagetable）和 handle page fault 这两大块的内容。
  • 第一步是page fault的处理
    • xv6 有三种不同类型的page fault，分别是 Instruction page fault, load page fault ,store page fault -> execute ,read,write
    • 对应到lazy allocation 我们只需要关心load page fault && store page fault,对应的是read && write operation
    • 查找risv手册可以得知，当r_cause的值为13 -> load page fault, r_cause=15 -> store pagefault,所以我们可以根据r_cause的值来判断page fault的类型
  • 第二步是观察page fault的地址
    • 我们只关心在myproc()->sz范围的page fault，超出范围的page fault 是正常情况，我们无需额外处理
    • 处理完page fault之后如何重新执行引起page fault的指令呢，这时候我们需要观察stval && sepc的值,我们知道sepc存放的用户程序当前的pc值，而stval则是引起page fault的虚拟地址。

具体实现

综合上述所说 实现过程如下：（xv6的用户stack在exec的时候分配）

  1. 判断r_cause的值来进入对应的处理分支
  2. 判断当前产生page fault的虚拟地址 是否在假装分配好的用户空间大小内(也即 stval <= myproc()->sz)
  3. 如果处于当前的进程空间范围，则为其申请空间，如果不在当前进程空间或者申请空间失败则 杀死该进程。
  4. 还有就是修改uvmunmap和uvmcopy使得测试通过，
    • 因为在原始的xv6里面 当父进程fork出子进程的时候，会将父进程p->sz大小的页全部copy到子进程，但是在lazy allocation的时候，父进程并没有这么多的页需要copy，而uvmcopy一旦发现（要copy的页）里面的pte是非法的，就会panic，所以会导致程序运行失败，因此我们需要把这里的panic 给去掉，直接continue，保持和父进程一致即可。
    • 其次则是uvmdealloc这个函数会调用uvmunmap，当需要回收用户程序的内存的时候会调用uvmdealloc这个func，然而umvunmap会检测该页是否真的存在，如果不存在的话就会panic，而在lazy allocation中，我们先alloc 在dealloc其实是没有用到任何页的，也就说这时候就会进入panic的状态，因为我们本来就没有 alloc page ，因此 这里也需要修改成continue，跳过该页。
  5. 修改walkaddr，使得在查找页表的时候也可以申请新的物理页 从而满足copyin/copyout/loadseg等函数的需求
    • 当我们需要copyin的时候，我们是直接从用户页表上copy数据到内核空间，但是当一个合法的虚拟地址并且使用malloc为其分配物理页的时候，这个虚拟地址实际上是非法的(由于lazy allocation)，因此我们需要在walkaddr的时候为虚拟地址分配物理内存。
