在阅读新版的时候，有很多容易被遗漏的点需要加强注意一下，以及算是读书之后的总结，所以这篇blog，会处于比较hypothesis的阶段。记录方式大概是以Q&A的形式

Q1: 为什么不将fork 和exec 两个system call 结合到一起，形成一个新的类似于windows的createProcess的system call？

  • A1 : 这样做的原因是因为有一定的历史遗留问题，其次这样做的好处是容易实现I/O重定向。当fork和file descriptor结合，子进程copy父进程的file descriptor table，并且共享文件的offset。

Q2:进程之间为什么需要强隔离性，以及xv6是如何实现强isolution的？

  • A2 : 因为我们不希望一个illegal的process去破坏其他正常的process的运行，如果任意一个进程都可以访问所有的物理地址的话，那么任意一个process都可以去恶意地修改别的进程的数据，因此我们需要对进程进行强隔离。类unix的操作系统的强isolution是通过页表来实现的，每个process拥有属于自己的private pagetable，这个页表对其他进程来说不可见，当一个程序的image被load进内存的时候，这个process的pagetable就已经被build 好了。process只被允许访问属于自己的进程空间的地址(一旦访问的虚拟地址没有对应的物理地址->也就是不存在该virtual memory address到物理地址的映射， 就会产生page fault)

Q3 :常见的privileged instruction有哪些？

  • A3 : disable/enable interrupts(开关中断) reading/wrting register that holds the address of pagetable(读写satp寄存器) 等等

Q4 : 如何理解进程这个概念

  • A4 : the unit of isolation in xv6 is a process(书上对process的定义)，其实在我看来进程就是对computer硬件资源的一个抽象。传统意义上，在过去80年代的批处理器时期，每台电脑在一段时间内只能处理一个进程，每当一个进程结束之后，才切换到下一个进程执行。在现在multi-process的computer上，我们可以认为每一个process都拥有着专属于自己的cpu，memory(即使这台电脑可能只有一个cpu)，也就是说当process运行的时候，在process的角度来看:cpu资源就被ta独享了，memory资源也是被ta独享的。更具体而言，process就是一个struct ，里面包含了这个process所对应的stack/内核栈/页表/状态/优先级，每当这个process被调度的时候，satp寄存器就会指向这个process的页表地址。然后可能会做一些初始化的动作，跳到main函数一条指令一条指令的执行下去。

Q5 : 使用页表的好处

  • A5 : 1.保持procees的pagetable 对其他进程不可见 2. 页表的虚拟地址是连续的，然而物理地址可以是不连续的 3.物理地址可以多次映射到不同的user pagetable (例如trampoline)

Q6 : 三种情况导致中断正常指令流，然后将控制权交给kernel

  • A6 : 1. 调用system call 2.异常 3.设备中断 这三种情况可以用trap概括
