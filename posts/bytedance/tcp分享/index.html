写一个真实可与外界通信的tcp协议栈

什么是cs144

cs144 是斯坦福大学 ee/cs 本科生大二的一门计算机网络课，在这门课的 Project 中，你将用 C++ 循序渐进地搭建出整个 TCP/IP 协议栈，实现 IP 路由以及 ARP 协议，最后利用你自己的协议栈代替 Linux Kernel 的网络协议栈和其他学生的计算机进行通信，非常 amazing！

TCP协议解决了什么问题

学过计算机网络的我们知道， 因为网络延迟/路由转发等各种原因 网络层无法保证packet的有序到达 ，而 TCP协议 是一个工作在传输层的可靠数据传输的服务，它能确保接收端接收的网络包是无损坏、无间隔、非冗余和按序的。后续会讲TCP协议是如何确保数据包的可靠传输的

tcp协议怎么样解决这些问题的

  • Fast Retransmit (快速重传）

  • 有序接收

  • Timeout Retransmit （超时重传）

  • 慢启动/ 线性启动

搭积木的第一步 搭建一个可靠的字节流（bytestream）

简单来说就是我们需要实现一个可以支持一些读写操作的buffer，这个buffer需要支持

  • 可以确定的容量 capacity

  • 一个指针指向读或写的位置

  • 一个标识 确保是否已经到了buffer的底部 _eof_flag

  • 一个int 标识 已经从buffer读了多少字节

  • 一个int 标识 已经往buffer写了多少字节

搭建积木第二步 如何重组乱序到达的packet（实现一个StreamReassembler）

让我们看看一个packet在buffer中不同的形态

  • 蓝色部分代表已经被上层应用所读取（如http/ftp/ssh）等等

  • 绿色的部分代表存放在bytestream中的 已经有序的字节流

  • 红色部分代表存放在一个缓冲区的 还未被组装的字节流，等待有新的packet到来会触发一次组装

  • Capacity 由 bytestream的大小和 unassembled_bytes（）的大小共同组成

std::map<key,value>

cpp中关于map的实现底层是一种称之为红黑树的结构，这里不展开关于红黑树的各种细节讨论，我们只需要知道红黑树是内部有序的，而且查找key和插入key的过程是Log(n)级别的。由于lab中多次涉及查找和插入过程，所以std::map提供了优秀的时间和空间复杂度。

如何处理到达的segment push_string()的编写

这个是streamAssembler中需要解决的最关键的问题，也是最难的问题，没有之一。因为你需要解决各种问题，一不小心就会面向testcase编程(就是一开始的我没错了) 下面我列出了一些需要应对的场景

  • 当segment的index 大于_cur_index + _capacity 直接丢弃

  • 当segment的index 小于_cur_index + _capacity 但是index + data.size() > _cur_index + _capacity 意味着我们可以接受该segment位于接受窗口内的部分，舍弃掉超出接受窗口外的部分！这一个很关键，是一个会卡住大多数人的点(包括我)，要仔细思考才能解决

  • 当remain_capacity 小于要data.size() 的时候 只写入remain_capacity大小的字符串，可以考虑用string.substr()方法

    • 对于index的处理
    • **index < _cur_index 重复的segment 但是可能存在data.size() > _**cur_index - index也就是这个segment的后面部分是我们之前没有接收到的
    • index == _cur_index (excepted segment) 这正是我们需要的segment，处理起来也很简单，就往bytestream里面写就完了，但是写完之后要对_queue里面的segment进行查看有无可以join进来的部分(找到需要重组的部分 然后拼接进bytestream)
    • index > _cur_index 乱序的segment 这个部分会难一点所以我编写了一个unorder_segment_handle的函数来处理

搭积木第五步 抽象一个tcp的发送方 （TCP_sender）

作为一个sender，关心以下的问题

  • 跟踪已经发送但是未被确认接收的segment

    • 这一部分其实已经很简单了，我只是顺带着说一下，用一个queue就可以完成这个工作，只需要在适当的时候push 和pop就ok了。那什么是适当的时候呢 ？

    • PUSH : 每当你发送需要consume sequence number的segment的时候(SYN/FIN/DATA)都需要push到队列里面

    • POP : 每当你接受到的ackno大于之前的ackno的时候，你需要check一遍queue里面有没有过期的segment(也就是已经被接受的segment)

    • 二进制指数退避 这个很简单就是当前的rto *2 就完事了，但是有个注意的点就是_window_size等于0的时候是不需要back off的，也就是说当_window_size = 0，保持当前的rto就可以了 （所谓TCP拥塞控制）

1. CLOSED -> SYN SENT *status* = *connect*表示发起连接
三次握手的第一个阶段，由client发出syn segment ;
2. LISTEN -> SYN RECEIVED  server 收到 SYN      segment,三次握手的第二个阶段，由server 回复syn-ack segment;
3. SYN SENT -> ESTABLISHED
    client 收到SYN-ACK segment,回复ACK segment;
    三次握手的最后一个阶段
4. SYN RECEIVED -> ESTABLISHED
    sever 收到ACK segment ，不回复;三次握手结束 Connection Established
5. SYN SENT -> SYN RECEIVED
    这是指双方同时想要建立连接的特殊情况，双方都发送了syn segment 并且都回复syn-ack segment。都进入SYN RECEIVED的状态，注意此时只会建立起一条connection，而不是两条。(TCP 做了特殊处理)
6. ESTABLISHED -> FIN WAIT 1
    收到application的close signal，发送 fin segment;(可以携带data)  Active Close       
7. ESTABLISHED -> CLOSE WAIT
    收到client端的fin segment，回复 ACK segment 进入close wait状态，可以继续发 data segment 
8. CLOSE WAIT -> LAST ACK
    收到application的close signal，发送fin segment，进入LAST ACK 状态
9. FIN WAIT 1 -> FIN WAIT 2
    client端收到sever端的ack segment 不回复任何segment 进入FIN WAIT 2
10. LAST ACK-> CLOSED
    sever收到client端对fin 的ack segment 进入 CLOSED
11. FIN WAIT 2 -> TIME WAIT
    cilent收到server端的fin 回复ack segment 进入 TIME WAIT(2Max Segment Lifetime)
12. TIME WAIT -> CLOSED
    经过2MSL的时间，进入CLOSED

  • 黄色数字解读
    • 在client端发送fin之后没有收到ack前，收到sever端的fin segment，进入CLOSING
    • client端收到sever端对自己的fin segment 的ack segment，直接进入TIME WAIT
    • client端收到fin-ack segment直接TIME WAIT.

关于TIME WAIT 以及为什么通常由client端来Active Close

关于为什么有TIME WAIT的解释 以及TIME WAIT存在会导致什么问题

  • TIMEWAIT导致的问题 同时主动关闭很多socket 会使很多socket处于TIME WAIT(2MSL 按照RFC规定是4mintes)，导致没有更多的socket/port 可以使用，性能严重浪费；可以指定REUSA_ADDR来解决

  • TIMEWAIT存在的必要

    • The first is to prevent delayed segments from one connection being misinterpreted as being part of a subsequent connection.防止delay的segment被另外的connection错误地接收。尽管可能性十分的低
    • The second reason for the TIME_WAIT state is to implement TCP’s full-duplex connection termination reliably. If the final ACK from end point 2 is dropped then the end point 1 will resend the final FIN. If the connection had transitioned to CLOSED on end point 2 then the only response possible would be to send an RST as the transmitted FIN would be unexpected. This would cause end point 1 to receive an error even though all data was transmitted correctly. 确保发送相应对方fin semgent的ACK segment可以被接受到，如果timeout了 sever会重发 fin segment，重复上述过程。
    • If the connection had transitioned to CLOSED on end point 2 then the only response possible would be to send an RST as the retransmitted FIN would be unexpected. This would cause end point 1 to receive an error even though all data was transmitted correctly. 我觉得这一点也很关键。TIMEWAIT存在的意义,如果client发送ack之后直接closed，sever端假设丢失ack，重发fin segment，这样会导致client 回复RST segment ，这样的后果真的太难顶了。
