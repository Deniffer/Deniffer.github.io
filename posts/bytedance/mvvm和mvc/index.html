关于对MVC和MVVM的理解

Reference Blog

其实之前MVC这种软件设计模式一直没有什么深刻的理解，包括到现在也没有说认识的很全面，虽然也没有应用过，但现在也总结一下我了解的MVC && MVVM MVC wiki && MVVM wiki

MVC（Model-View-Controller) 其实在我看来更多是对数据的操作（controller） 展示数据的视角（view） 以及数据存储（Model） 的隔离设计，这三者是必然是相互联系，但为了可复用（reusable），等特性 我们需要对不同的部分进行分隔。

  • Model：只关心数据是如何存放的 不关心如何使用/展示(Code that cares about how data is stored)

  • View : 只关心数据是如何展示的（Code that cares about how data is displayed）

  • Controller: 只关心数据是怎么被创建/更新/删除的 （Code that cares about how data is stored and how it is displayed）

由于MVC的实现多变，很难统一（有很多种不一样的MVC实现），所以这里贴了维基百科的MVC标准，从而来解释我眼中标准的MVC

其实整个MVC的设计最重要的是组织代码结构（在我看来），使得更多的代码部分可以被复用。在客户端的MVC大概是这样的，用户触发事件（点击屏幕/按按钮等）Controller接收到事件之后，对Model的数据进行操控 更新，当Model更新了某些数据时，需要通知View来展示用户操作的有效性（也就是用户在点击屏幕之后 ，屏幕要发生反应之类的），这里一般会用到观察者模式，在Model里面注册View的监听。观察者模式总结

Tips: 在绝大多数的 MVC 架构模式中，模型都不会主动向视图或者控制器推送消息；模型都是被动的，它只存储整个应用中的数据，而信息的获取和更新都是由控制器来驱动的。

一个MVC各部分的例子

  • Model part ：在实际开发过程中，当我们存储date的时候 ，是用秒（second）的形式来记录的（从UNIX Time 1970/01/01 00:00 开始计算，比如到现在2022/01/25 经过了多少个second（number of second），也就是一个超大的整数， 比如到今天就是1643076004 seconds ）

  • View part ：当我们展示给用户是另一种presentation 一般的 date format 是 yy/mm/dd 这样的形式的.

  • Controller part： 我们如何进行转换(second -> Date Format)就是Controller的part.

如上所述，各个部分的职能是分隔开来的。

MVP（Model-View-Presenter）

  • 首先MVP是MVC的一个变种，在MVP中，Model不和View通信，而是通过Presenter代理，所有从View发出的事件都会通过代理给Presenter进行处理，同时presenter也通过View的接口与其通信。
    • MVP有两个不同的变种，一个是Passive View 另一个是Supervising Controller

      • Passive View

      • 在Passive View中，View层是被动的，他不会改变自己的任何状态，所有的状态都是由Presenter来间接改变的。 如前端的Html和CSS 只负责View层的结构和内容，本身不具备任何的逻辑。由此带来的好处是View层不依赖与其他任何层级，这也就最大化了View层的可测试性，同时也将View层和Model层进行了合理的分离，两者不再相互依赖。

      • Supervising Controller

      • 简单来说Supervising Controller 就是将允许部分简单的同步的操作，可以由Model直接更新。在该模型下，View层接管了部分简单的视图逻辑，由Model直接同步更新，而Supervising Controller仍需要负责响应用户输入和一部分更复杂的视图逻辑、模型状态同步的工作。对于用户输入的处理，监督控制器的做法与标准 MVP 中的 Presenter 完全相同；但是对于视图、模型的同步工作，监督控制器会尽可能地将所有简单的属性以数据绑定的形式声明在视图层中。好处是一些简单的逻辑不需要再通过Supervising Controller，坏处是增加了View和Model的耦合，带来了一定的复杂度。

关于MVVM

提到MVVM 必须先提到PM（Presentation Model）

  • PM模型

  • PM 模式将View中的全部状态和行为放到一个单独的Presentation Model（展示模型）中，协调领域对象（Model）并且为视图层提供一个接口。

    • PM 通过引入Presentation Model 将Model层的数据与复杂的业务逻辑（一般来说在Controller）封装成属性与数据同时暴露给视图，让View和Presentation Model的属性进行同步。

      • Presentation Model 包含所有View渲染所需要的动态信息，包括View的内容（Text、color）、组件启用情况、除此之外还会将某些方法暴露给View用于事件响应。

      • Presentation Model 对于Model层的操作以及为View层暴露接口都是非常容易的，在整个 PM 模式中，最为麻烦的就是View和****Presentation Model 状态的同步。

      • 因为Presentation Model 是View的抽象，其中包含了View的状态（属性）和行为（动作），视图的行为可能很少发生改变，但是视图状态的改变就是非常常见的了，那么同步视图和展示模型的代码应该放哪里就是一个需要考虑的问题了。

      • 到目前为止，我们能够放置状态同步代码的地方其实只有两个，也就是视图和展示模型；如果将同步的代码放在视图中，那么可能会影响视图的测试，不过由于现在的大部分客户端程序完全没有测试，这一点其实也影响不大；如果将代码放在展示模型中，实际上就为展示模型增加了视图的依赖，导致不同层级之间的耦合。

<Window x:Class ="WPFDataBinding.MainWindow" Title="MainWindow" Height="350" Width="604">
   <Grid>
      <Label Name="nameLabel" Margin="2">_Name:</Label>
      <TextBox Name="nameText" Grid.Column="1" Margin="2"
         Text="{Binding Name}"/>
      <Label Name="ageLabel" Margin="2" Grid.Row ="1">_Age:</Label>
      <TextBox Name="ageText" Grid.Column="1" Grid.Row ="1" Margin ="2"
         Text="{Binding Age}"/>
   </Grid>
</Window>

在 WPF 中我们可以使用 Binding 关键字在 XAML 中完成双向绑定，当 TextBox 中的文字更新时，Binder 也会更新 ViewModel 中对应属性 Name 或者 Age 的值。也就是说PM中最难解决的问题 被我们放在View层的XAML中，通过隐式的方式实现了状态的同步

  • MVVM 和PM的贡献

    • 为关键的是Presentation Model/ViewModel创建了一个View的抽象，将View中的状态和行为抽离出一个新的抽象，这才是 MVVM 和 PM 中需要注意的。
