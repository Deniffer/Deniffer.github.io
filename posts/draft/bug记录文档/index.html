记录工作过程中所遇到的Bug

身为一名iOS开发工程师，在程序开发的时候不可避免遇到各种各样的Bug，

  1. 有一些Bug 我们可以很轻易地解决
  2. 有一些Bug 我们可能需要stackOverFlow之后，经过一定的“调试”之后解决
  3. 有一些Bug 我们依靠自己并不能解决

在我不长的工作时间里面，我发现真正消耗我很多时间的是第二类的Bug， 第一类Bug 往往都比较明显，通过简单的断点调试 抑或是lldb的 po someObject.xxxProperty就可以fix，解决耗时常在1min - 15mins内。 第三类Bug出现的原因大多数是因为对业务不够熟悉导致的问题，抑或者是从第二类Bug 解决时间太长，严重影响到正常的开发进度之后的一次升级。这种情况虽然耗费的时间也是比较长，但是第三类Bug往往出现的次数比较少，并且随着对业务的熟悉程度/对语言的熟悉程度上升 也会逐渐退化成第二类Bug。

而第二类Bug，我粗浅的分类为

  1. 一些情况下是对于一些API的使用不够熟练
  2. 一些情况下是对这门语言本身的一些机制不够熟悉
  3. 一些情况下是因为业务不熟悉

对于第一种情况，我只能说十分常见，因为有太多的API 需要去学习，我们不可能使用所有API，那么对API不熟悉是一种很常见的情况。在这种情况下，我首先会看这个API的调用方，看看别人是怎么用的，对于我的场景是否适用（就抄嘛，太简单了）。其次，如果发现调用API的时候发现，不太符合预期，可以考虑看看Apple Document 看一下官方解释（一般情况下，可能有用），到了这一步一般也还不能fix问题，我们就需要看看有没有别人和我们遇到一样的问题了。于是乎我们求助于强大的Google，一般如果是别人遇到过的问题，都会有专门的stackOverFlow的帖子，这个时候加以分析，选择一个适用自己场景的answer即可。 如果不幸运，你遇到的问题别人没有遇到过（极其低的概率，以及有可能你搜索的方向可能不正确），这个时候，不得已你需要了解这个API真正的实现，以及上下游的依赖。于是乎，这不可避免将耗费你的一些光阴。

这也正是为什么我要写这篇文章的原因，很多时候，你fix了一个问题，ok这个过程短暂地停留在你的脑海里一段时间，在过后的某一个时间段里面，你有极大的可能会遇到同样的问题或者是类似的问题， 于是乎，你又重复地进行了一遍上述的一个过程，这其实是一种低效地解决Bug的方法，就像是中学时期我们用过的“错题本” 一般，这篇文章的定位也不外乎如此，更高效地解决重复的问题。

UI类Bug

1. UIScrollView 滚动 不及预期 。预期中ScrollView的滚动能到视野范围外

原因是因为scrollView的subView依赖了除scrollView的其他View做自动化布局导致的。 这个Bug 出现了两次，两次都是同样的原因，由于subView依赖了scrollView的parentView 所以导致在滑动的时候，这些理因滚动到可视区域外的subView依然坚挺在可视区域内。 我个人的归因是因为Apple 内部View布局的依赖关系导致的。

2. UI看不见或者说不符合预期

首先是尝试用LookIn去看UI的布局，

  1. addSubview有没有生效，subView的size是不是为(0,0), x,y坐标是不是出错了，这些信息LookIn都可以很准确的告诉你。
  2. 其次，如果是LookIn有对应的subView 并且subview的frame 符合预期，但是用肉眼无法看到对应的视图，可以看看资源的字符串是否拼写错误导致资源加载不了。
  3. 如果是通过IESLiveReMaker && IESLiveMasUpdate 修改UI的布局，没有生效的话， 由于masUpdate只能更改已有的属性，所以这是一个值得注意的”坑“。
  4. 如果是通过reMaker来进行布局，却发现UI的布局没有改变， 这个 可以先考虑把所有对该UI的布局收敛到一个函数内，外部所有对这个UI的修改都通过这个函数进行。 如果还没有解决问题的话， 那么可以认为你看的View 可能并不是你想要修改的View，到了这一步已经不是布局的问题了，是更深层次的Bug导致了本次问题的发生。

3. RAC绑定 多走或者是少走

RAC在第一次绑定的时候，就会触发一次Block。如果想避免这种情况的话，可以用skip的操作跳过绑定是的block执行。 如果发现RAC绑定之后，里面的Block回调并不会被触发，这个时候第一时间要考虑的是，RAC绑定的Object是不是已经被释放了，也就说RAC bind nil这种情况，自然里面的Block回调不会被触发。

4. UIScrollView 不响应滑动事件

有几个可能，1 scorllEnable设置错误 2.没有设置delegate=self 但排查之后这两个原因都不是

排查了很久也找不到 相关的原因

5. UICollectionViewCell 使用cellForIndexPath return nil

cellForIndexPath return nil 的原因 从官方注释的原因是

Returns any existing visible or prepared cell for the index path. Returns nil when no cell exists, or if index path is out of range.

我检查无数遍我的index 根本没有超出range，所以一定是这个过程中出现了某些问题， 一开始我想尝试换一个api，比如说-dequeueReusableCellWithReuseIdentifier:forIndexPath: 但是按照我的理解，这个是拿cell pool 里面可以复用的cell 应该不太对， 所以还是坚持应该是我用的方法不对的原因。 于是我开始看NSIndexPath的结构 &&stackOverflow 可惜的是 stackOverFlow的答案并没有奏效

由于NSIndexPath的结构不太熟悉 导致了我这样的用法 [[NSIndexPaht alloc]initWithIndex:index] 按照道理来说没有啥大的问题，但是当我打印看collectionView 的IndexPath的时候，并不是我想象中的样子， 但是我又在NSIndexPath上找不到别的init方法， 于是就陷入了尴尬的局面，我知道我应该要构造怎么样的NSIndexPath，但是我不知道怎么用init构造。再然后我就开始想象，这个问题应该别人也遇到过，聪明的我就开始全局搜索NSIndexPath的使用，果然 这个case 别人也同样遇到了，但是它使用的是[NSIndexPath indexPathForItem: inSection:] 然后我换了API就解决了这个问题。 这个API是NSIndexPath的一个category， 位于NSIndexPath+UIKitAdditions.h的文件里，由于我只浏览了NSIndexPath.h 并没有看其有没有额外的category，所以排查时间比较久。 && 我没有尝试[NSIndexPath ..] 而是上来就是 [[NSIndexPath alloc] ..]，导致我的补全提醒 没有帮上忙， 不然的话应该很快就能解决问题。

6. UITableView 的一些时机问题

遇到一个问题是想要在tableView 的delegate执行完之后，对visibleCell 做埋点操作。但是发现就算使用了layoutIfNeeded/reloadData 也解决不了问题， 其原因是因为我在执行这些函数的时候，superView的frame还没有被set，导致了这个问题。只有在superView的frame确定之后才会触发对应的layoutSubView。这个问题最根本的原因是在于对整体的UIView /viewController 各个阶段该干什么事情还是不够清晰。没有总结一套UI生命周期/渲染周期的结论，所导致的问题。 还有UIView对应的某些Properties也不是很熟悉最终导致了这个问题。

7. 多线程removeAllObject的问题

这个问题是在和其他横向团队合作的时候依赖他们的代码的时候出现的，因为是crash 所以xcode给的提醒很迷惑人 是对应的[LivecoreLayerInfo copyWithZone] can not find selector for instance in 0x0028f2ff2 ..类似的提醒， 但是看堆栈却发现crash在TTNetWork Manager，当时感觉是xcode 的堆栈出现了问题，就没有继续往堆栈的方面深思，转而去考虑出现问题的那个函数。因为尝试了打断点去复现这个问题，但是发现打断点之后这个问题不复现了， 只有在断点被删除后，才复现了问题，于是我们猜想可能是多线程竞争的问题。 紧接着我们又尝试了把锁的范围扩大，但发现还是会crash，直到有其中一位同学说，会不会有可能是因为上报埋点导致的错误，因为只有上报埋点才有可能crash在TTNetworkManager 这里。听他这么一说，我觉得十分有道理，然后开始分析就算是上报埋点也不会crash才对。然后再仔细一看对应的代码,类似于下图

业务相关/业务不熟悉导致的Bug

1. Dependency Injection 的Object 拿不到

表现为有对应的函数调用，但是没有应有的效果。 这个时候先考虑，

  • 在该函数内能不能拿到DI，
  • 其次是生命周期， 在拿DI的时候，这个对象的生命周期是否已经结束，如果已经结束的话，同样是拿不到DI。一个典型的场景是在callBack 的时候

2. 时序导致的问题

这个问题已经是比较强业务相关的问题， 比较好的方法是通过打日志来看问题的具体表现。

3. PB生成的字段

如果是server没有下发的情况下，得case by case 去看， 如果字段是object 的话，没有下发则是一个空对象 而不是nil， 如果是assign类型的话， server没下发则是0 记住 你虽然打印出来的是nil 但是其实是0 (因为用po去打印的话是错误的 所以会出来nil)

4. Timer 取消的地方

每写一个timer的时候都需要考虑，有哪几个时机需要将其释放 是一个好的习惯

记录Code Review 自己需要过的点

很简单嘛，code review 减轻别人的负担，也减轻自己的负担

  1. 开关的合理性，尽量覆盖开关开和关的情况
  2. 记得判断异常情况 ，比如网络请求失败，callback释放对应的权限，减法判断结果大于零
  3. Weakify && strongify 在callback的时候
  4. 三元表达式兜底 改成btd方法
  5. tableView 记得register
  6. 放置的地方是否合理
  7. 函数空行 && 换不换行取决当前文件的风格 && 是否多一个空格
  8. 直接让server下发整段文案
  9. 清屏状态判断
  10. 删掉注释代码
  11. 注意离屏渲染
  12. 写接口的时候 冒号用tab对齐下，{ 前加空格
  13. copy/strong 如果是可变对象则用strong。不可变对象则用copy
  14. 字典的话 直接addEntry
