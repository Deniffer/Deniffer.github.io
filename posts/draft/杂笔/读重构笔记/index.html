读《重构-改善既有代码的设计》

When to reFactor？

书中给了一个很好的定义：当你想要在一个模块添加一个新的feature，但是你发现你如果想要实现这个新feature很困难的时候， 你可以考虑先重构整一个模块，而后再往其上添加你的feature。

The First Step In Refactoring

重构的第一步拥有是先写完善的testcase， 这样做的是为了尽可能避免重构之后模块出现bug的情况，只要test case够多，你就越有自信，你重构之后的模块是完备的，是没有Bug的。

The Second Step In Refactoring

将某些复杂，臃肿的函数抽出来，将其拆分成为更小的模块，使函数的逻辑更加内聚。 有一个很好的思想是：一个函数只做一件事情，尽量避免使一个函数大而全， 一个好的函数代码行数应该是在1-40行以内， 当你发现一个函数大于40行， 那么你可以考虑将其拆分成多个部分来实现。其次函数名尽量体现了这个函数的行为，而不是有额外的逻辑。

  • 举个例子 一个login function 尽量避免 setupSession的操作。

别人能不能光从API 就能看出来这个函数到底是干嘛用。 最后 每当你进行一小部分的重构时 ，你就需要跑一遍testcase 以确保这一部分的代码是没有错误的， 当test case 不过的时候，因为改动范围有限，很快就能定位Bug的位置并将其修复。

’’’ Any fool can write code that a computer can understand. Good programmers write code that humans can understand. ’''
